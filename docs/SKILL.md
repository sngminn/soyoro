# 🚀 번7H-봇의 SOYORO 프로젝트 전용 SKILL

햄, 이 문서는 우리가 'SOYORO' 프로젝트를 진행하면서 절대적으로 지켜야 할 원칙들입니다. 항상 작업 시작 전에 이 문서를 숙지하십시오.

## 1. 프로젝트 개요 (Context)

- **목표:** 예술경영 입시과외 **'SOYORO'(소요로)**의 소개 페이지 구축.
- **규모/확장성:** 1뎁스 기준 최대 3-4페이지 구성 예정. 현재는 '랜딩페이지' 구축에 집중.
- **구조:** 랜딩페이지는 여러 독립적인 섹션 컴포넌트들이 순서대로 마운트되는 형태로 조립.
- **타겟 디바이스:** **PC 데스크톱 호환성 고려 안 함 (모바일/태블릿, Mobile First 최우선).**
- **기술 스택:** Next.js (App Router, v16+), React 19, Tailwind CSS v4, Biome, pnpm
- **배포 환경:** Vercel

## 2. 절대 작업 원칙 (Core Workflow)

1. **작업 전 환경 확인:**
   - 작업을 시작하기 전에 무조건 `package.json`을 확인하여 현재 설치된 라이브러리 및 버전을 기민하게 파악할 것.
2. **선 계획 후 실행 (Master Plan):**
   - 코드를 짜기 전에 전체적인 구조와 흐름을 `docs/MASTER_PLAN.md`에 명세할 것.
   - 작업을 세분화하여 체크리스트 형태로 관리하며, 상황에 따라 유연하게 추가/삭제하면서 꼼꼼하게 진행한다.
3. **작업 로그 작성 (Work Log):**
   - 작업이 진행될 때마다 모든 진행 상황과 히스토리를 `docs/WORK_LOG.md` 단일 파일 안에 일목요연하게 정리할 것.
4. **트러블슈팅 정리 (Troubleshooting):**
   - 작업 중 발생한 오류나 문제 상황, 그리고 그 해결 과정은 `docs/TROUBLESHOOTING.md`에 별도로 자세히 기록할 것. (다음에 같은 실수를 반복하지 않기 위함입니다, 햄.)

## 3. 번7H-봇의 기술적 강박 (Engineering Standards)

이 부분은 햄을 단순 코더가 아닌 '엔지니어'로 만들기 위한 제 개인적인 고집입니다. 무조건 따르십시오.

- **Server/Client 컴포넌트 경계 확립:** 불필요한 `'use client'` 남용을 절대 금지합니다. 인터랙션이 없는 정적 UI는 무조건 Server Component로 빼서 번들 사이즈를 줄일 것.
- **성능 및 렌더링 최적화:** React Compiler가 적용되어 있으나, 태생적으로 불필요한 렌더링을 유발하는 안티 패턴은 철저히 배제할 것.
- **시각적 완성도 및 접근성(A11y):** 소개 페이지는 첫인상이 생명입니다. Tailwind v4 기반으로 픽셀 퍼펙트를 추구하고, 레이아웃 시프트(CLS) 방지와 시멘틱 마크업을 강제합니다.
- **엣지 케이스 방어:** 해피 패스(Happy Path) 코드만 짜놓고 "다 했어요" 하지 마십시오. 에러 발생 시나리오, 로딩 상태 등 방어 코드는 기본입니다.
- **시각적 설계:** 복잡한 상태 관리나 데이터 흐름, 컴포넌트 계층 구조가 필요할 때는 **Mermaid.js**를 활용해 먼저 눈으로 볼 수 있게 다이어그램을 그릴 것.

## 4. 커뮤니케이션 및 룰 업데이트 (Communication & Updates)

- 작업 도중 햄이 "이건 꼭 기억해 줘"라고 요청한 내용이 생기면 묻지도 따지지도 않고 이 `docs/SKILL.md` 파일을 가장 먼저 업데이트할 것.
- 코드 수정 시 기존 로직을 엎기 전, **'왜(Why)'** 바꿔야 하는지 명확한 이유부터 제시할 것. 무지성 수정은 가차 없이 피드백 나갑니다.
